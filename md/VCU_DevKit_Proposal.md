# VCU DevKit — 汽车VCU软件开发辅助工具方案

> **版本**: v0.1 Draft | **日期**: 2026-05-13 | **定位**: 内部团队桌面端工具

---

## 1. 背景与痛点

当前VCU软件开发依赖Vector工具链（CANoe / dSPACE / ETAS INCA），存在以下效率瓶颈：

| # | 痛点 | 现状 | 影响 |
|---|------|------|------|
| P1 | AUTOSAR SWC配置繁琐 | 手动在DaVinci/EB中定义SWC、Port、Runnable、Interface | 一个VCU项目200+ SWC，配置耗时2-3周，易出错 |
| P2 | CAN矩阵 ↔ 代码脱节 | DBC更新后需手动同步信号映射、打包/解包代码 | 信号遗漏或映射错误导致实车问题 |
| P3 | 诊断服务配置易错 | UDS服务、DTC、Snapshot手动配置，无校验 | 诊断覆盖率不达标，售后排查困难 |
| P4 | 标定参数无统一管理 | 参数散落在A2L、Excel、模型中 | 版本混乱，标定团队与开发团队信息不对称 |
| P5 | 需求-代码追溯断裂 | 需求在飞书多维表格，代码在Git，模型在dSPACE | 影响ASPICE审核，变更影响分析靠人工 |
| P6 | 测试用例编写低效 | HIL测试用例手动编写，重复度高 | 测试覆盖率不足，回归测试周期长 |

## 2. 工具定位

```
┌─────────────────────────────────────────────────────────┐
│                    VCU DevKit                            │
│                                                          │
│   "Vector工具链的效率放大器，不是替代品"                   │
│                                                          │
│   不重新造轮子 ──→ 补齐Vector工具链之间的断层              │
│   不替代CANoe ──→ 自动化CANoe中重复的手动操作             │
│   不替代INCA  ──→ 让标定参数的管理前置到开发阶段           │
└─────────────────────────────────────────────────────────┘
```

**核心原则**：
- **读写Vector原生文件格式**（DBC/ODX/A2L/ARXML），零迁移成本
- **自动化重复劳动**，人只做决策不做搬运
- **渐进式引入**，每个模块独立可用，不强制全量切换

### 技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| 框架 | **PyQt6 / PySide6** | Python生态丰富，AUTOSAR/XML解析库成熟，团队易上手 |
| ARXML解析 | `autosar` / `lxml` | 原生支持AUTOSAR 4.x ARXML |
| DBC解析 | `cantools` | 开源成熟，支持DBC/ARXML/ODX互转 |
| A2L解析 | `asammdf` + 自研 | 解析ASAP2标准A2L文件 |
| ODX解析 | `odxtools` | ODX 2.2/3.0解析 |
| 模板引擎 | `Jinja2` | 代码生成、报告生成 |
| 数据库 | **SQLite** | 轻量，单机部署，参数/追溯数据本地存储 |
| 版本管理 | 内置Git集成 | 配置文件版本化 |

## 4. 核心模块设计

### 4.1 SWC Designer — AUTOSAR SWC可视化配置

**解决问题**：P1（AUTOSAR SWC配置繁琐）

**功能**：

```
┌─────────────────────────────────────────────────────────┐
│ SWC Designer                                            │
│                                                         │
│  ┌─────────────┐    ┌──────────────────────────────┐   │
│  │ SWC树形列表  │    │  SWC详情编辑区                │   │
│  │             │    │                              │   │
│  │ ▼ VCU       │    │  SWC: VCU_PowerMgmt          │   │
│  │   ├ Power   │    │  ┌─────────────────────┐    │   │
│  │   ├ Drive   │    │  │ <<ServiceProvided>>  │    │   │
│  │   ├ Thermal │    │  │  Port: PowerStatus   │    │   │
│  │   ├ Diag    │    │  │  Interface: IPower   │    │   │
│  │   └ Comm    │    │  └─────────────────────┘    │   │
│  │             │    │                              │   │
│  │ [模板库]     │    │  Runnables:                  │   │
│  │ ▼ Power系   │    │  ├ RE_PowerOn (10ms)        │   │
│  │ ▼ Drive系   │    │  ├ RE_PowerOff (10ms)       │   │
│  │ ▼ Diag系    │    │  └ RE_FaultHandler (event)  │   │
│  └─────────────┘    └──────────────────────────────┘   │
│                                                         │
│  [从ARXML导入] [导出ARXML] [生成骨架代码] [一致性检查]    │
└─────────────────────────────────────────────────────────┘
```

**核心能力**：
- **可视化SWC拓扑**：拖拽式定义SWC、Port、Interface、Data Type
- **模板库**：预置VCU常用SWC模板（PowerMgmt / DriveControl / ThermalMgmt / DiagMgmt等）
- **ARXML双向同步**：读取现有ARXML → 可视化编辑 → 回写ARXML（兼容DaVinci/EB配置）
- **骨架代码生成**：从SWC定义自动生成C代码骨架（Runnable函数签名、RTE调用接口）
- **一致性检查**：Port接口类型匹配、Runnable周期校验、数据类型兼容性

**与Vector生态对接**：
- 导出的ARXML可直接导入DaVinci Configurator
- 支持读取EB Tresos生成的ARXML
- 两套工具链的ARXML差异通过适配层兼容，用户可选择目标工具链

### 4.2 CAN Builder — DBC驱动的通信配置与代码生成

**解决问题**：P2（CAN矩阵 ↔ 代码脱节）

**功能**：

```
┌──────────────────────────────────────────────────────────────┐
│ CAN Builder                                                  │
│                                                              │
│  ┌── DBC文件管理 ──────────────────────────────────────────┐ │
│  │  VCU_Main.dbc  v2.3  │  Diff: +3 signals, ~1 signal    │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌── 信号矩阵 (可编辑表格) ────────────────────────────────┐ │
│  │  Msg Name    │ Signal        │ Start│ Len │ Factor │ Off │ │
│  │  VCU_Status  │ VCU_PowerMode │  0   │  4  │  1     │ 0   │ │
│  │  VCU_Status  │ VCU_Ready     │  4   │  1  │  1     │ 0   │ │
│  │  VCU_Status  │ VCU_SOC       │  8   │  8  │ 0.5    │ 0   │ │
│  │  VCU_Torque  │ Tq_Request    │  0   │  16 │ 0.1    │ -500│ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌── 生成产物 ─────────────────────────────────────────────┐ │
│  │  ☑ CAN Pack/Unpack代码 (C)                               │ │
│  │  ☑ 信号-变量映射表 (.h)                                   │ │
│  │  ☑ CANoe CAPL节点 (.can)                                 │ │
│  │  ☑ 信号变更报告 (.xlsx)                                   │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  [打开DBC] [版本Diff] [生成代码] [同步到CANoe] [导出报告]      │
└──────────────────────────────────────────────────────────────┘
```

**核心能力**：
- **DBC可视化编辑**：表格化展示Message/Signal，支持批量编辑
- **版本Diff**：DBC新旧版本对比，精确到Signal级别，标记新增/修改/删除
- **代码自动生成**：
  - C代码：Pack/Unpack函数、信号枚举、物理值换算
  - CAPL：CANoe测试节点代码
  - 信号映射头文件（SWC可直接引用）
- **变更影响分析**：DBC变更后自动标记受影响的SWC和测试用例
- **校验规则**：信号重叠检测、字节序一致性、取值范围合理性

- **变更追踪**：记录每次参数修改的来源、原因、影响范围
- **与INCA对接**：生成的A2L可直接导入ETAS INCA

## 6. 项目实施路线

### Phase 1：最小可用（10周）

> CAN Builder + SWC Designer 并行开发，Core Engine先行

| 模块 | 范围 | 产出 |
|------|------|------|
